# Архитектура F.R.I.D.A.Y

## Идея

F.R.I.D.A.Y — персональная ИИ-ассистентка. Пользователь делегирует ей задачи,
она планирует, напоминает, формирует документы, шлёт отчёты и сама решает
поставленные задачи. Основной интерфейс — Telegram; управление — через сайт.

## Компоненты

| Компонент | Роль | Технология |
|-----------|------|------------|
| Сайт + бэк-офис | Регистрация, личный кабинет, подписки, админка | Next.js (App Router) |
| Telegram-бот | Основной интерфейс ассистентки | grammY (этап 2) |
| База данных | Память и центр данных FRIDAY | Supabase (Postgres + Auth) |
| Движок ИИ | «Мозг» ассистентки | Claude (Anthropic API) |
| Хостинг бота | Постоянно работающий сервер | Railway |
| Хранение кода | Репозиторий | GitHub |

## Монорепозиторий

```
apps/web      — сайт, личный кабинет, бэк-офис (есть)
apps/bot      — Telegram-бот (этап 2)
packages/*    — общий код: типы БД, движок, команды (этап 2)
supabase/     — SQL-миграции схемы БД
docs/         — документация
```

## Модель данных (этап 1)

- **profiles** — пользователи (роль: `user` / `admin` / `superadmin`).
- **subscriptions** — подписка пользователя (статус, срок действия).
- **bots** — Telegram-бот пользователя: токен, `owner_telegram_id`, статус.
- **payments** — заявки об оплате; подтверждаются администратором.
- **audit_log** — журнал действий бэк-офиса.

Доступ к данным защищён политиками Row Level Security: пользователь видит
только свои строки, администратор — все.

## Роли и подписки

- **superadmin** — владелец проекта. Полный доступ к бэк-офису.
- **admin** — помощник по бэк-офису (при необходимости).
- **user** — обычный пользователь.

Оплата — ручная: пользователь переводит деньги и оставляет заявку в кабинете,
администратор подтверждает её в бэк-офисе → подписка активируется на
оплаченный срок (продление складывается с остатком).

## Безопасность: привязка бота к владельцу

Главное требование. Бот FRIDAY должен реагировать **только** на Telegram ID
владельца, указанный в поле `bots.owner_telegram_id`.

Правила для бота (этап 2):

1. На каждом апдейте сверять `message.from.id` с `owner_telegram_id`.
2. Чужой `from.id` — молча игнорировать (бот «не активен» для него).
3. Пересланные сообщения (`forward_from` / `forward_origin`)
   **никогда** не считать новым пользователем — учитывать только реального
   отправителя `from.id`.

Это закрывает утечку из первой версии бота, когда сообщение с чужого аккаунта
переключало бота на новый чат и раскрывало личные данные.

## Дорожная карта

- **Этап 1 — сайт + бэк-офис** *(готово)*
  Регистрация, личный кабинет, подключение бота, подписки, админка.
- **Этап 2 — ядро бота**
  grammY-бот на Railway, привязка к владельцу по ID, движок Claude,
  базовые команды (`/plan`, `/task`, `/report`, …), напоминания.
- **Этап 3 — продвинутые возможности**
  Генерация файлов (PDF/DOCX/XLSX), генерация и оценка изображений,
  динамическое создание команд, автономные задачи, интеграции
  (Trello, Notion).
